About the Project

Background:

After building my first encryptor application in Java, I wanted to deploy it to my portfolio. Even though it would be technically possible to deploy this application using the Angular framework that this site is built on, I knew that this would not be a security best practice for an encryption application. I therefore decided to host the application on a backend NodeJS server in order to protect the logic from being accessed by the user.

Operation:

To encrypt text, enter the text into the “Enter Plaintext” textfield and click on the “Encrypt" button. The encrypted cipher text will appear in an output field that replaces the project information.

To decrypt cipher text, enter the cipher text into the “Enter Ciphertext” text field and click on the “Decrypt" button. If the ciphertext is valid, the decrypted plain text will appear in an output field that replaces the project information. If the cipher text is invalid or an error occurs, an error message will appear instead of the decrypted plain text.

Technical Details:

The encryption and decryption processes both work by sending GET http requests to a custom-built API that performs the respective operations and returns the results or error messages. I built the API using NodeJS with Express and hosted it on a Vercel stateless server. Special care was taken to ensure the NodeJS server is reusable and scalable including building it in an MVC architecture and following best practices in directory architecture.

This JAVA console application encrypts text inputted by the user and returns cypher text using the AES algorithm. Upon startup the user selects either encryption or decryption mode. In encryption mode the user enters the text to be decoded, followed by a secret key which is used to encrypt the text. The ciphertext is then produced and displayed in the console. The Initialization Vector, IV, is appended to the ciphertext returned. It is not necessary to keep the IV secret.

In decryption mode, the user enters the ciphertext which includes the appended IV along with the secret key. The IV is first unappended from the ciphertext. The decryption key is then generated using the value entered for the secret key. The ciphertext is then decrypted using the generated key and the IV. The original plaintext is then returned. Manipulation of the IV is handled automatically from the backend and requires no additional operations by the user.
